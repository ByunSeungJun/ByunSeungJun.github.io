02 Production Incident ★
재시작 악순환에 빠진 프로덕션 장애를 구조로 종결 — DB 커넥션 풀 고갈
ProjectFine Play — 실서비스 운영 인프라 (대회 트래픽 급증기)
역할진단·설계·조치   시점2026.02 트래픽 급증기
HikariCPMySQLDockerAWS 다인스턴스
GOAL
재시작 루프로 전 인스턴스가 다운됨

운영 중 Too many connections가 나면서 앱이 못 뜨고, Docker가 자동 재시작하며 커넥션이 더 쌓이는 악순환으로 전 인스턴스가 내려갔습니다.

ACTION
풀 상한을 전체 인스턴스 합 기준으로 재설계
  • Threads_connected가 MySQL 한계(151)에 닿았고 재시작마다 계단식으로 늘었습니다. 원인은 HikariCP 풀 상한이 없던 것이었습니다.
  • 먼저 전 인스턴스 컨테이너를 강제 종료해 커넥션을 정리하고 재시작 루프를 끊었습니다.
  • 풀 상한은 인스턴스 수 × 상한이 DB 한계를 넘지 않게 역산했습니다(4×10=40, 한계 151). idle·max-lifetime도 함께 조정했습니다.
RESULT
커넥션 150+ → 40~50개로 안정

한계 대비 여유를 두고 안정 운영에 들어갔고, 재시작 악순환은 다시 나오지 않았습니다.

LEARNING
자원은 인스턴스 하나가 아니라 전체 합으로

오토스케일·재시작 환경에서는 풀 상한을 개별 인스턴스가 아니라 전체 인스턴스 합으로 잡아야 한다는 걸 이 장애로 확인했습니다.

인시던트 흐름
커넥션 수 — 조치 전후 (정성 도식)
역산 — 인스턴스 4 × 풀 상한 10 = 40 ≤ MySQL 151. 클러스터 전체 커넥션을 DB 한계 아래로 고정했습니다.

03 Auth · Cache
Redis 기반 인증·캐시 설계 — 토큰 회전과 장애 격리
ProjectFine Play — 통합 JWT 인증 시스템
역할인증·캐시 설계·구현
RedisJWTRTR분산락
GOAL
토큰이 탈취돼도, Redis가 죽어도 버티게

토큰이 탈취·재사용돼도 피해를 막고 싶었고, 동시에 Redis가 멈춰도 인증과 서비스가 함께 죽지 않게 하고 싶었습니다.

ACTION
상태를 저장하고, 장애를 흡수하도록 설계
  • 리프레시 토큰 회전(RTR) — 토큰마다 상태(VALID·USED·REVOKED)를 Redis에 두고, 이미 쓴 토큰이 다시 오면 재사용으로 보고 해당 유저의 토큰을 일괄 무효화했습니다.
  • 동시에 들어오는 회전 요청은 setIfAbsent 분산락으로 하나만 통과시켰습니다.
  • 로그아웃한 액세스 토큰은 남은 만료시간만큼 TTL을 걸어 블랙리스트에 넣어 자동 소멸시켰습니다.
  • Redis 장애 시 캐시 실패를 흡수해 인증·프록시가 계속 동작하도록(graceful degradation) 했습니다.
RESULT
토큰 재사용 대응 + 단일 장애점 제거

재사용 공격에 대응하는 인증 구조를 갖췄고, Redis가 죽어도 서비스가 멈추지 않게 만들었습니다.

LEARNING
보안은 “장애일 때”까지 설계해야 한다

정상 동작만이 아니라, 장애가 났을 때 무엇을 열고 무엇을 닫을지(페일오픈/클로즈)를 함께 정해야 한다는 걸 배웠습니다.

리프레시 토큰 회전 (RTR)
동시성 · 장애 처리
  1. 1분산락 (setIfAbsent)동시 회전 요청을 하나만 통과시켜 경쟁 상태 차단
  2. 2블랙리스트 TTL로그아웃 토큰을 잔여 만료시간만큼만 보관해 자동 소멸
  3. 3Graceful degradationRedis 장애 시 캐시 실패를 흡수, 인증·프록시는 계속 동작

04 Performance ★
리스트 조회 성능 최적화 — 3N 요청을 N+1로
ProjectFine Play — 앱 유저 목록 · 어드민 대시보드 조회
역할성능 개선
JPAMySQL지연 로딩캐시
GOAL
리스트가 커질수록 요청이 폭증했다

유저 리스트가 길어질수록 화면 하나에 네트워크 요청이 크게 늘었습니다. 동작은 그대로 두고 비용만 줄여야 했습니다.

ACTION
중복 호출 제거 + 지연 로딩 + 공용 캐시
  • 유저 카드마다 프로필 이미지·즐겨찾기를 개별 호출해 최대 3N 요청이 났습니다. 목록 응답에 이미 있는 값을 재사용해 per-user 호출을 없앴습니다(3N→N+1).
  • 백엔드 대시보드의 N+1은 @ManyToOne을 즉시 로딩에서 지연 로딩으로 바꿔 제거했습니다.
  • 공용 이미지 컴포넌트로 디스크 캐시(1000개·60일)와 표시 크기 디코드 제한을 약 45곳에 일괄 적용했습니다.
RESULT
리스트 요청 3N → N+1 감축

DTO를 보강하면 →1까지 줄일 수 있음을 설계상 확인했고, 정적 분석 0 error를 유지했습니다.

LEARNING
성능은 셀 수 있는 지표부터 줄인다

요청 수처럼 명확한 지표로 개선을 먼저 검증하고, 이어서 응답시간을 실측해 체감 성능까지 확인합니다.

리스트 조회 요청 수 — Before / After
Before3N카드(=유저)마다 프로필 이미지·즐겨찾기를 개별 호출
AfterN+1목록 응답에 있는 값 재사용 · DTO 보강 시 →1까지
그 밖의 조치
  1. 1@ManyToOne 지연 로딩 전환대시보드 조회의 N+1 쿼리 제거
  2. 2공용 이미지 캐시 약 45곳디스크 캐시 1000개·60일 + 표시 크기 디코드 제한
  3. 3캐시 누수 차단검색·즐겨찾기 provider를 autoDispose로 전환

05 Transaction
트랜잭션 전파 설계 — 무엇을 같은 커밋에 둘지
ProjectFine Play — 결제·구독 도메인
역할트랜잭션 경계 설계·디버깅
Spring TxJPAMySQL
GOAL
너무 많이 묶인 트랜잭션이 데이터를 삼켰다

하나의 트랜잭션에 일을 너무 많이 묶으면, 뒤에서 난 예외가 앞의 성공까지 되돌립니다. 어긋나면 안 되는 데이터를 지켜야 했습니다.

ACTION
같은 커밋에 둘 것과 분리할 것을 나눔
  • 구독 강등 처리가 상위 트랜잭션 롤백에 휩쓸려 유실되거나 재청구가 중복되던 경로를, 후처리를 REQUIRES_NEW로 분리해 독립 커밋되게 했습니다.
  • “결제는 성공했는데 커밋 후 후처리 예외로 500이 나던” 문제는, 커밋 이후 단계(알림·외부 호출)를 트랜잭션 밖으로 빼 원인을 없앴습니다.
  • 무엇을 같은 커밋에 두고 무엇을 분리할지를 기준으로 트랜잭션 경계를 다시 그었습니다.
RESULT
강등 유실·중복 청구·성공 후 500 제거

롤백에 데이터가 휩쓸리는 경로가 사라졌고, 결제 성공 뒤 500이 재발하지 않았습니다.

LEARNING
다 묶는다고 안전한 게 아니다

트랜잭션은 크게 묶을수록 안전한 게 아니라, 같은 커밋에 둘 것과 분리할 것을 나누는 설계라는 걸 배웠습니다.

트랜잭션 경계 — 묶기 vs 분리
핵심 — 정합성이 필요한 일은 같은 커밋에, 외부 호출·알림은 커밋 밖으로. 트랜잭션 경계를 나누는 게 곧 정합성 설계입니다.
