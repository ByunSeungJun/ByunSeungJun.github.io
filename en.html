Ways of Working

“Deliberate 50% less, execute 10% more (50210)” — I'd rather build small and learn fast than polish a perfect plan. The less precedent a problem has, the more I set my own hypothesis and bar, then let real users and data set the direction.

  • Test the most uncertain assumption first, in the smallest way — and judge on metrics with visible cause and effect, not vanity numbers. Performance gets cut from real measurements; incidents get driven to their root cause so they can't recur.
  • Remove repetition through structure and automation. I run an AI agent (Claude Code) in parallel sessions — build · code review · security audit — and keep an ADR (architecture decision record) practice so decisions and their context become team assets.
  • Tools get delegated, judgment doesn't — only well-defined tasks go to agents; spec definition, output review, and the final call stay with me.
